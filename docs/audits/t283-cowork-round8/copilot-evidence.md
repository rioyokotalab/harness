## Critique

- **Finding confirmed reciprocally, single-scoped.** Both traces agree: this is a receipt byte-identity/provenance defect, not a bypass of the unreachable driver-held seal. The import decision is bound entirely to read #2. I endorse the driver's refusal to attach broader confinement or authorship claims.
- **`O_NONBLOCK` — I keep it; the driver's Critique omits it. This is a real divergence, and I judge it required, not scope creep.** The refactor moves the regular-file test from *before* the content access (today's `lstat`-first `require_owned_kind`) to *after* the `open` (`fstat`). That reordering reintroduces a vector the current code does not have: a same-UID FIFO/device leaf makes a plain `O_RDONLY` `open` block indefinitely, hanging import. `O_NONBLOCK` makes the `open` return promptly so `fstat` can reject the non-regular file with the existing message; it is a genuine no-op on the regular file that then passes `fstat`. Because the risk is *created by the single-descriptor shape itself*, the flag is intrinsic to that shape, adds no schema/CLI/doc surface, and keeps the refactor from being strictly weaker than the baseline on a special-file leaf. (A reviewer could instead keep an `lstat` pre-gate, but that reintroduces the very second path op we are removing; `O_NONBLOCK` is the cleaner way to stay single-descriptor.)
- **`MAX_SEAL_BYTES` — I withdraw it; the driver is right to reject it.** I proposed a 64 KiB cap in the independent stage. On reflection it is out of the frozen charter ("no unrelated hardening") for two concrete reasons, not just style: (1) it does nothing for byte identity — the single read already binds parse-bytes and digest-bytes by construction; (2) today's `read_text`/`read_bytes` are both unbounded, so a fixed cap is a *behavior change* that would refuse a currently-accepted large valid seal — a latent regression, not hardening. Read the descriptor to EOF (chunked loop until empty) to match current behavior exactly, with no cap. I therefore reject my own item.
- **Diagnostics must be preserved deliberately, not incidentally.** Two exact messages are easy to lose in the rewrite: (a) `json.loads` on raw *bytes* auto-detects encoding and would surface a non-UTF-8 seal as a `JSONDecodeError`, collapsing the distinct `seal is not valid UTF-8` message — so decode explicitly (`data.decode("utf-8")` → `UnicodeDecodeError` → existing UTF-8 message) *before* `json.loads`; (b) the symlink-leaf message currently comes from `S_ISREG`-on-`lstat`; after the change it must come from mapping the `open` error `errno in (ELOOP, EMLINK)` to the existing "must be real and not a symlink" text, with `ENOENT` → "missing protocol entry", and every other non-regular/uid/hard-link case reproduced from `fstat` with the current strings. Existing symlink/hard-link/uid/malformed/non-UTF-8 assertions then match unchanged.
- **Deterministic regression: prove the reopen is gone, do not race it.** My independent draft floated a "wrapper that pauses" to drive a real #2→#3 swap; I withdraw that — it is effectively a production test hook and the charter/driver disfavor it. The correct, fully deterministic proof is *structural*: after the refactor there is no third read at all, so byte identity is guaranteed by construction and there is nothing to race. Assert that (i) `import_copilot` no longer contains `session_path(args.seal).read_bytes()` and holds no second seal-path read, (ii) `load_seal` opens with `O_NOFOLLOW` and returns a `(value, digest)` pair consumed directly into `receipt["seal_sha256"]`, and (iii) a happy-path import yields `receipt.seal_sha256 == sha256(seal-bytes)`. Structure + happy-path + the preserved edge tests cover it without instrumentation.
- **Fixed vs residual, stated precisely.** *Fixed by the descriptor-bound read:* the #2↔#3 byte-identity gap (no second read exists) and the former #1→#2 symlink-follow window (`read_text` followed symlinks; `O_NOFOLLOW` + `fstat` on one inode does not). *Explicit remaining residuals, unchanged and to be documented, not claimed fixed:* (1) a same-UID replacement of the leaf *before* `open` — the attacker's bytes are then both validated and digested, but they must still satisfy every structural and cross-session check (`stage_manifest_sha256` ↔ live `stage.json`, roles, mode, phase, `destination_before`), and since seal content is already co-pilot-writable in a same-UID model this grants no new capability and the *decision* is unchanged; (2) external-location/parent-path confinement — `seal_location_ok` still resolves only the parent, and the fix makes no claim about stopping a co-pilot who can reach the driver-held seal. Cross-file atomicity is out of scope.
